## üì© 1. MESSAGE FORMAT RULES

All system messages must follow these standards.

### **1.1 Core Principles**
- One message = one idea.
- Short, clear, no filler words.
- Neutral and professional tone.
- End every sentence with a period.
- No emojis, no exclamation marks, no unnecessary punctuation.

### **1.2 Color Codes**
| Purpose | Code |
|---------|------|
| ‚úÖ Success | `&a` |
| ‚ùå Error / Denial | `&c` |
| ‚ÑπÔ∏è Info / Help | `&f` |

**Only one color per message.**

### **1.3 System Errors vs User Errors**
There are two types of errors:

| Error Type | Example Scenarios | Message Format |
|------------|--------------------|----------------|
| **User-caused (Intentional)** | No permission, missing item, invalid command, cooldown | Specific: `&cNo permission.` |
| **System/Internal/Code Error** | Exceptions, API failure, null pointer, backend crash | Generic only: `&cAction failed.` |

**System errors must never reveal technical reasons, stack traces, or internal details.**

‚úî Allowed:
```
&cAction failed.
&cOperation failed.
&cRequest failed.
```

‚ùå Not allowed:
```
&cNullPointerException on line 54.
&cDatabase connection failed.
&cAttack failed because target was null.
```

### **1.4 Message Format Templates**

| Type | Template |
|------|----------|
| Success | `&a<Action completed>.` |
| Error | `&c<Action failed>.` or `&cNo permission.` |
| Info | `&f<Instruction or neutral message>.` |

### **1.5 Placeholder Rules**
- Use `{placeholder}` only when needed.
- Must be lowercase: `{user}`, `{item}`, `{value}`, `{from}`, `{to}`
- No `< > ( ) [ ]` or quotes around placeholders.

Examples:
```
&aItem {item} added.
&cItem {item} not found.
&aOwnership transferred from {from} to {to}.
```

## ‚öôÔ∏è 2. CODE & PROJECT STRUCTURE RULES

This section defines how code must be structured, written, and organized.

### **2.1 Code Style Philosophy**
- Extremely simple and minimal.
- As little code as possible without losing functionality.
- No redundant, unused, or duplicated logic unless it keeps things simpler.
- Focus on performance: minimal CPU, RAM, and tick usage.
- Runtime performance is more important than faster loading times.
- Readable and direct code is preferred over abstraction and complexity.

### **2.2 AI Usage Rules (If Using AI for Code)**
- AI must only generate what is specifically requested.
- No extra features, utils, abstractions, or improvements unless explicitly asked for.
- No assumptions, no ‚ÄúI thought this would be helpful.‚Äù
- If it isn't requested, it must not be created.

### **2.3 File & Class Structure Rules**
- **Each command must have its own file.**

Examples:
```
kitcreate.java
clanchattoggle.java
claninvite.java
```

- **Group related commands in folders/packages:**
```
/kits       ‚Üí all kit commands
/clans      ‚Üí all clan-related commands
/chat       ‚Üí chat-related commands
```

- **Keep file names simple and readable.**
  - ‚úÖ `usermanagement.java`
  - ‚ùå `UserManagementHelperServiceUtil.java`

### **2.4 Minimize Utils / Events / Mixins**
- Only create an event, util, or mixin when absolutely necessary.
- Avoid large utility classes with random functions.
- Prefer keeping logic inside the command file ‚Äî even if it means repeating a few lines.
- Simplicity is more important than abstraction.

### **2.5 Performance Rules**
- Use as little CPU and RAM as possible.
- No pointless loops, complex data structures, or expensive calculations.
- Avoid unnecessary schedulers, timers, tick-based checks.
- Only use async or multithreading when required.
- Avoid heavy reflection, streams, and caching unless needed.

### **2.6 Forbidden Practices**
‚ùå No over-engineered class names like `PlayerEntityInteractionManagerService`.  
‚ùå No unused or commented-out junk code.  
‚ùå No giant utility or helper classes for simple actions.  
‚ùå No unnecessary design patterns (factories, builders, dependency injection).  
‚ùå No excessive events or mixins when simple direct method calls work.
